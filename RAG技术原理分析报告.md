# RAG 技术原理分析报告

## 1. RAG 架构概述

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将外部知识库检索能力与大语言模型生成能力结合的问答架构。它不是完全依赖大语言模型自身参数中的知识，而是在回答前先从知识库中检索相关资料，再把检索结果作为上下文交给大语言模型生成答案。

本项目面向“云南非物质文化遗产”领域，使用本地文档构建知识库，通过 BGE 中文 Embedding 模型生成向量，使用 FAISS 作为向量数据库，并调用 Qwen 大语言模型完成最终回答。

## 2. Indexing：索引阶段

索引阶段的目标是把原始知识文档转换成可检索的向量索引，主要流程如下：

1. 文档加载：读取 TXT、PDF、Word、Markdown、HTML 等格式文件，提取正文内容。
2. 文本清洗：过滤空文档和过短文本，保留文档来源、文件名、文件类型等元数据。
3. 文本分块：将长文档切分为多个 Chunk。本项目默认使用 `chunk_size=500`、`chunk_overlap=100`，既控制单个片段长度，也保留上下文连续性。
4. 向量编码：使用 `bge-small-zh-v1.5` 模型将每个 Chunk 转换为 512 维语义向量。
5. 索引构建：对向量做 L2 归一化后写入 FAISS `IndexFlatIP`，用内积近似余弦相似度检索。
6. 元数据保存：保存 Chunk 内容、来源、文件名、Embedding 和创建时间，便于后续展示检索来源。

## 3. Retrieval：检索阶段

检索阶段负责从知识库中找到与用户问题最相关的文档片段。

本项目流程如下：

1. 将用户问题编码为 Query Embedding。
2. 对 Query 向量做 L2 归一化。
3. 使用 FAISS 从向量索引中召回候选 Chunk。
4. 使用 MMR（Maximal Marginal Relevance）进行检索优化，在相关性和多样性之间取得平衡。
5. 设置相似度阈值，如果最高分低于阈值，则提示知识库中没有足够相关资料。
6. 返回内容、来源文件、相似度分数，供界面展示和 Prompt 构建使用。

MMR 的优势是可以避免返回内容高度重复的多个片段，使最终 Prompt 覆盖更多信息点。

## 4. Generation：生成阶段

生成阶段将检索结果、历史对话和用户问题组装成 Prompt，然后交给大语言模型生成回答。

本项目 Prompt 主要包含：

1. 角色设定：云南非物质文化遗产知识库智能问答助手。
2. 历史对话：保留最近多轮问答，支持上下文连续追问。
3. 参考资料：包含检索到的 Chunk、来源和相似度。
4. 用户问题：当前输入的问题。
5. 回答约束：优先依据参考资料回答，不编造，资料不足时说明原因，并在末尾列出来源。

这种方式可以让模型回答更贴近知识库内容，同时提高答案可追溯性。

## 5. Embedding 模型对比

| 模型 | 特点 | 优点 | 局限 |
| --- | --- | --- | --- |
| bge-small-zh-v1.5 | 中文语义向量模型，维度较小 | 本地运行快，占用资源低，适合课程实验和中小型知识库 | 表达能力弱于大模型 Embedding |
| bge-large-zh | 更大的中文向量模型 | 语义表达能力更强，检索精度通常更高 | 推理速度较慢，占用内存更高 |
| text2vec-base-chinese | 常见中文句向量模型 | 使用简单，中文支持较好 | 在复杂语义检索上可能不如 BGE 系列 |
| OpenAI / Qwen Embedding API | 云端向量模型 | 效果稳定，维护成本低 | 需要联网和 API 成本，数据需上传到云端 |

本项目选择 `bge-small-zh-v1.5`，主要考虑中文领域文档、本地离线运行和实验环境资源限制。

## 6. 向量数据库对比

| 向量数据库 | 特点 | 适用场景 | 局限 |
| --- | --- | --- | --- |
| FAISS | 本地高性能向量检索库 | 单机实验、中小型知识库、快速原型 | 不自带完整服务端、权限管理和分布式能力 |
| Chroma | 易用的本地/轻量级向量数据库 | 课程实验、原型系统、LangChain 集成 | 大规模生产能力有限 |
| Milvus | 分布式向量数据库 | 企业级大规模知识库、海量向量检索 | 部署和维护复杂度较高 |

本项目使用 FAISS，因为它安装简单、检索速度快，适合本地课程作业展示。

## 7. RAG 对缓解 LLM 幻觉的优势

RAG 可以缓解大语言模型幻觉，主要体现在：

1. 提供外部事实依据，使模型不只依赖参数记忆。
2. 可追溯来源，用户可以查看答案来自哪些文档。
3. 知识更新成本低，只需更新知识库和索引，不必重新训练模型。
4. 可限制回答范围，适合企业知识库、规章制度、产品手册等场景。

## 8. RAG 的局限

RAG 不能彻底消除幻觉，主要局限包括：

1. 检索失败会导致模型缺少正确上下文。
2. 文档切分不合理会破坏上下文完整性。
3. 检索结果相关但不充分时，模型仍可能过度推断。
4. 知识库本身如果过期或错误，答案也会受影响。
5. Prompt 约束不强时，模型可能忽略资料边界。

因此，实际系统需要结合合理切分、重排序、置信度阈值、来源展示和人工审核机制。

## 9. 本项目实现总结

本项目完成了一个完整的 RAG 知识库问答系统，包含多格式文档加载、文本分块、Embedding 生成、FAISS 向量索引、MMR 检索优化、Prompt 构建、Qwen API 问答、多轮对话管理、文档上传更新、检索来源和置信度展示。整体符合课程实验中“智能问答系统与 RAG 知识库助手”的核心要求。
